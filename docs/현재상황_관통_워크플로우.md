# inframon — 모든 것을 관통하는 워크플로우 (현재 상황)

> **이 문서의 목적**: inframon 프로젝트의 *목표 → 시스템 골격 → 개발 전략 → 실데이터 파이프라인 →
> 현재 위치 → 게이트 → 다음 액션* 을 **하나의 줄기**로 꿴다. 세부는 각 전문 문서로 안내한다.
>
> **기준 시점**: 2026-06-30 · 테스트 **286개 수집 green** · 4대 엔진(CV·InSAR·PINN·FRAM) 전부 real+핫스왑 등록.
>
> **한 줄 현황**: 소프트웨어는 합성·실데이터 선별까지 **완성·검증**. 남은 건 (1) 실 SLC 충분량 + WSL2 처리로
> 실 변위 시계열 1회 산출, (2) 실 제원·GNSS·손상라벨로 정확도 게이트 G3~G5 실수치 확정, (3) 타 팀(VLM·Bmaps).

## 1. 목표 — KAIA 안에서 inframon이 책임지는 것

```
[우리 — inframon] ━━━━━━━━━━━━━━━━━━━━━━━━━┓        [타 팀 — VLM]
 InSAR 변위 → PINN 가상센싱 → 통일 데이터 패키지 ━╋━→ + CAD + 정밀안전보고서
   (수직·종축)   (전체변위·EI·성분)  (CSV·JSON·figures) ┃         ↓
                                                        ┃   대한민국 시방서·법규 평가
                                                        ┃         ↓
                                                        ┃   지식그래프 → 보수보강 추천
```

- **우리 책임**: 위성 변위 추출 + 구조해석 + VLM 이 삼킬 **통일 데이터**. 안전 최종판정은 VLM+시방서(타 팀).
- **1차년도 목표**: InSAR+PINN 을 Bmaps 에 통합 — **소프트웨어 완성**, 실데이터 관통만 진행 중.
- **만드는 것은 "프로그램"** — SARvey 기반 InSAR 처리를 *프로그램 자체가 end-to-end* 로 수행. (남의 연구
  데이터/스택에 piggyback 하지 않음. 검증은 합성·데모 데이터로.)

## 2. 시스템 골격 — 불변의 척추 + 교체 가능한 엔진

```
┌────────────────────────────────────────────────────────────────────┐
│ 데이터 흐름(불변):   CV ──▶ InSAR ──▶ PINN ──▶ FRAM ──▶ 경보(CRI)   │
│ 통로(불변):          contracts/  project.h5  (/cv /insar /pinn /fram)│
│ 피드백:              FRAM.critical_members ──▶ CV/InSAR 점밀도 상향  │
└────────────────────────────────────────────────────────────────────┘
```

- **계약은 성역**(`contracts/`, Pydantic + project.h5): 모듈 간 *유일* 통로. 엔진은 이 계약을 깨지 않고
  내부만 stub→real 로 교체. 배열 레벨 검증(`contracts/array_schema.py`)으로 형상·dtype·차원심볼(N/M) 일관성까지 검사.
- **핫스왑**(`orchestrator/engines.py`): `--engine X=real` 로 엔진 하나만 켜고 나머지는 stub.
- **골든 회귀**(`tests/test_golden_regression.py`): 고정 시드 산출물의 계약·수치를 박제 → 조용한 회귀 차단.
- ⚠️ **절대 보존 계약**: `V_func_series_ds[4,M]` 행 순서 `(thermal, load, bearing, foundation)`. `los_ds` 단위는 **mm**.

### 4대 엔진 real 핵심
| 엔진 | real 핵심 | 파일 |
|---|---|---|
| **CV** | Otsu+CC / Transformer(SegFormer) 분할 → PCA 축선 → 형상증거 부재분할(SAM 폴백) → geo_transform/crs·미터 길이 | `cv/real_engine.py` |
| **InSAR** | Track H5 → CV 좌표 정합(world→pixel→world xyz·DEM z·고정단 거리) | `insar/real_engine.py`·`geotransform.py` |
| **PINN** | PyTorch PINN + 형식별 PDE + FEM 모달 + 절대 EI 식별 + 외생(온도·교통량) 구동 | `pinn/real_engine.py`·`pinn/pde.py` |
| **FRAM** | 점별 공명 + 함수망 N-K + isotonic 캘리브 + 6측면 함수망 + 예측발산 R_div + 경보보완 | `fram/{real_engine,network,calibration}.py` |

## 3. 개발 전략 — 왜 이렇게 짓는가

1. **계약 우선**: 먼저 데이터 계약을 고정(REAL·안정)하고, 엔진은 stub 으로 전 구간을 먼저 관통시킨다.
2. **핫스왑으로 한 개씩 교체**: stub→real 을 엔진 단위로 교체, 한 번에 하나만 위험에 노출.
3. **골든 회귀로 보호**: 교체할 때 계약/수치가 조용히 깨지면 FAIL. 의도된 변경만 `UPDATE_GOLDEN=1` 로 갱신.
4. **검증은 합성·데모로**: 실데이터가 없어도 전 구간이 동작·검증되도록. 실데이터는 *정확도 게이트* 확정에만 필요.

> 빌드 순서(완료): Phase1 인프라(핫스왑·골든) ✅ → InSAR real ✅ → CV real ✅ → PINN real ✅ → FRAM real ✅.

## 4. 관통 워크플로우 — 실 한 가닥으로 본 end-to-end

마커: ✅완료 · 🔄진행중 · ⏸대기 · 🔴외부의존

### A~E. InSAR 선별 — Windows · ✅ (다운로드·WSL2 불필요, 검색만)
지도 위치 → 교량 확인(OSM) → 위성 데이터 선별(ASF) → master 선정(ERA5) → 조건 게이팅 → SARvey 번들.
```bash
# 선별 + 조건 게이팅 + F 준비 번들
python -m inframon --insar-conditions  data/jeongjagyo_real
python -m inframon --make-sarvey-config data/jeongjagyo_real
```
산출: `data/jeongjagyo_real/{bridge_target,track_selection,selection_criteria,master_selection_era5,processing_manifest,sarvey_config}.json` · 교량 형식별 SARvey 세분(거더/트러스/아치/가동/케이블) 반영.

### B. 실 SLC 다운로드 — Windows · ✅ 5장 (배관검증용)
```bash
python scripts/download_slc.py data/jeongjagyo_real data/jeongjagyo_real/SLC --limit 5
```
정자교 상승궤도 5장(2024-01-07~02-24, 총 ~39GB) + 무결성 검증. ⚠️ SARvey 시계열엔 **15장+ 필요**.

### C. SAR 처리 (F) — WSL2/Linux · 🔄 (외부 도구 실행)
ISCE2·MiaplPy·SARvey 는 Windows 불가 → WSL2. `scripts/wsl_sarvey/` (실 granule 박힘).
```bash
cd /mnt/d/프로그램 ; RECIPE=data/jeongjagyo_real ; WORK=data/jeongjagyo_real
bash scripts/wsl_sarvey/00_setup_env.sh                 # 최초 1회 conda env
bash scripts/wsl_sarvey/10_download.sh   $RECIPE $WORK   # 궤도·DEM·aux (SLC 스킵)
bash scripts/wsl_sarvey/20_stack_isce.sh $RECIPE $WORK   # 코레지·간섭도  ◀ 진행 지점
bash scripts/wsl_sarvey/30_miaplpy.sh    $RECIPE $WORK   # phase-linking
bash scripts/wsl_sarvey/40_sarvey.sh     $RECIPE $WORK   # SARvey MTI
python3 scripts/wsl_sarvey/50_export_to_inframon.py --sarvey-h5 $WORK/sarvey/outputs/*_ts.h5 --out $WORK/track.h5
```
진행을 시각적으로: `python -m inframon --insar-progress data/jeongjagyo_real --recipe data/jeongjagyo_real --watch 5`

### D. 인제스트 + 4대 엔진 (핫스왑) — Windows · ⏸ track.h5 대기 (코드 완성)
track.h5 가 나오면 **이 아래는 자동**.
```bash
python -m inframon --check-track data/jeongjagyo_real/work/track.h5         # ready=0/not=1
python -m inframon --demo --insar-source data/jeongjagyo_real/work/track.h5 \
  --out data/jeongjagyo_real/project.h5 \
  --engine cv=real --engine insar=real --engine pinn=real --engine fram=real
```
CV(ROI·부재·지오) → InSAR(정합·변위) → PINN(전체변위·EI·성분) → FRAM(CRI·경보).

### E. KAIA 핸드오프 (CSV·VLM 패키지) — Windows · ✅
```bash
python -m inframon --export-csv data/disp.csv  --out data/jeongjagyo_real/project.h5 --bridge-id 정자교
python -m inframon --export-vlm data/정자교_vlm --out data/jeongjagyo_real/project.h5 --bridge-id 정자교 --zip
```
CSV(점×시점 롱포맷) + VLM 패키지(manifest·summary·narrative·figures·관측한계) · 통일 규약(mm·ISO·WGS84·부재 enum).

### F. Bmaps 통합 — ✅ (합의 대기)
```bash
python -m inframon --serve-api --registry data/bridge_registry.json --cors-origin https://bmaps.kict.re.kr
```
REST 엔드포인트 + CSV/VLM 다운로드 + 탭 예제(Leaflet·VWorld·React). 🔴 합의 3건: bridge_id 체계·지도 SRS·인증.

## 7. 지금까지 도출한 핵심 결론

1. **정자교는 단일 상승궤도로만 관측 가능** — ASF 실질의: 상승 path127 41장면, 하강 2장면뿐. asc+desc
   연직 분리 불가 → 단일궤도 LOS·종축만. (조건 G2)
2. **상승궤도로 열팽창(종축) 관측은 가능** — 축선 29°(OSM 절점 PCA) vs 상승 LOS, 종축 민감도 g=0.41
   (임계 0.30 초과 pass). bbox 종횡비 근사(g=0.13)는 틀린 경보였음. (조건 G1)
3. **변위 단위 1000배 버그 발견·수정** — `los_ds`는 계약상 mm 인데 Bmaps API 가 ×1000 → 과대보고. 제거.
4. **FRAM CRI 국소화 한계** — 단일궤도·강계절성분이 침하 국소화를 흐림(상관 −0.08). asc+desc 연직
   소비(opt-in) 시 −0.08 → +0.52 복원.
5. **5장으론 SARvey 시계열 불가** — phase-linking 은 ~15~20장 필요. 5장은 다운로드 인증 + ISCE2 스택·간섭도(20단계)까지의 배관 검증용.
6. **선별·조건·번들은 실데이터로 완전 동작** — 다운로드·WSL2 없이 ASF·OSM·Open-Meteo 실질의로 41장면 선별→조건 게이팅→번들 관통.
7. **교량 형식별 SARvey 차등이 실제로 다름** — 트러스/아치/가동이 거더 기준과 분기되어 config 로 흐름.
