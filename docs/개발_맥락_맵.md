## 1. 한눈에 보기

**inframon**은 InSAR·PINN·CV·FRAM 4대 엔진을 하나의 파이프라인으로 묶은 통합 교량 인프라 모니터링 플랫폼이다. 데이터 흐름은 단방향으로 고정되어 있다:

- **CV** — 위성 영상에서 교량을 탐지하고 부재를 분할하여 측정 영역(ROI)을 자동 산정
- **InSAR** — ROI 기반으로 측정점의 변위 시계열(LOS·종방향)을 추출
- **PINN** — 변위를 물리 성분(열팽창/하중/침하/이상)으로 분해하고 구조응답(처짐/변형률/응력/고유진동수)·물리 파라미터(EI, alpha)를 역산
- **FRAM** — 위 결과로 4종 공명 지표를 계산해 **공명 위험 지수 CRI**와 4단계 경보를 산출

**현재 상태** (요약은 [§0](#0-구현-종합-2026-06--전체-기능모듈도구)):
- 4대 엔진은 처음 전부 **STUB**였으나, **현재는 4대 모두 real 구현**이 추가돼 핫스왑(`--engine X=real`)으로 선택한다. stub 은 기본값으로 병존. (§4·§5·§6·§7)
- **골격·데이터 계약(contracts)은 REAL**이며 성역이다 — 엔진 내부만 교체하면 다운스트림은 영향받지 않는다.
- **실데이터 경로 가동**: 선별 A~E → (WSL2 F) → Track preflight(`--check-track`) → 인제스트(`--import-track-h5`/`--engine insar=real`) → 맞춤형 PINN(`--custom-pinn`). 검증은 합성/데모로(실 수치 게이트는 사용자 데이터 대기).
- 테스트 **194**, GitHub CI(ruff+pytest)·비공개 백업.

핵심 설계 철학: **엔진은 서로를 직접 호출하지 않고, 오직 contracts(Pydantic 모델 + 단일 `project.h5`)를 통해서만 데이터를 주고받는다.** 이 덕분에 4개 모듈을 독립적으로 개발하고, STUB를 mock 삼아 병렬 개발한 뒤 핫스왑할 수 있다.

## 2. 데이터 계약 — 시스템의 척추

모든 모듈 간 통신은 `src/inframon/contracts/`의 두 파일을 통한다.

- **`schema.py`** — 각 엔진의 출력 구조를 Pydantic 모델로 "박제"한다. 7개 모델(`CVGeometry`, `CVOutput`, `InSAROutput`, `PINNOutput`, `FRAMWarning`, `FRAMOutput`)과 2개 표준 상수 튜플(`MEMBER_TYPES`, `FRAM_FUNCTIONS`)로 구성.
- **`io.py`** — `ProjectStore`가 단일 `project.h5` 파일을 감싼다. `write_array`/`read_array`/`write_meta`/`read_meta`/`has_meta`/`require_group`을 h5py로 실제 구현.

### 계약이 작동하는 4가지 메커니즘

1. **`*_ds` 규약 (by convention)** — 큰 배열(`[H,W]`, `[N,M]` 등)은 절대 Pydantic 안에 직접 담지 않는다. 모델은 `roi_mask_ds='/cv/roi_mask'` 같은 **HDF5 데이터셋 경로 문자열만** 들고 다닌다. 따라서 모델은 가볍고 JSON 직렬화가 가능하다.
2. **디스크-경계 타입 검증** — `write_meta`는 `model_dump_json`, `read_meta`는 `model_validate_json`으로 왕복하므로 잘못된 스키마는 로드 시 Pydantic `ValidationError`로 실패한다.
3. **표준 라벨 단일 진실 소스** — `MEMBER_TYPES = ('deck','pier','abutment','bearing')`와 `FRAM_FUNCTIONS = ('thermal','load','bearing','foundation')`가 CV→InSAR→PINN→FRAM 전반에서 부재/기능 인덱스의 기준이다.
4. **흐름 의존성이 시그니처에 박힘** — `run_insar(store, cv)`, `run_pinn(store, insar)`, `run_fram(store, insar, pinn)`. 특히 PINN의 `V_func_series_ds`가 FRAM 공명 계산의 핵심 입력이고, `FRAMOutput.CRI_ds`가 최종 위험 지수다.

### project.h5 구조

```
project.h5
├── /cv     (attrs['meta'] = CVOutput JSON)   + 데이터셋(roi_mask, member_*, grid_density)
├── /insar  (attrs['meta'] = InSAROutput JSON) + 데이터셋(los, longitudinal, xyz, ...)
├── /pinn   (attrs['meta'] = PINNOutput JSON)  + 데이터셋(comp_*, strain, V_*, ...)
└── /fram   (attrs['meta'] = FRAMOutput JSON)  + 데이터셋(R_ij, amplification, CRI, ...)
```

> 주의: `contracts/__init__.py`는 0바이트 빈 파일이라 `from inframon.contracts import CVOutput` 같은 단축 임포트가 불가능하다. 모든 소비자는 `.schema` / `.io` 서브모듈 경로를 직접 사용한다.

## 3. 파이프라인 흐름

`orchestrator/pipeline.py`의 `run_pipeline`이 `ProjectStore(mode='w')` 컨텍스트 안에서 순차 실행한다:

```
PipelineConfig
   │
   ▼
run_cv(store, cfg) ──────────► CVOutput
                                  │ (roi_mask, member_label, geometry, grid_density)
                                  ▼
run_insar(store, cv, cfg) ─────► InSAROutput
                                  │ (los[N,M], longitudinal[N,M], xyz, member, dates, coherence)
                                  ▼
run_pinn(store, insar, cfg) ───► PINNOutput
                                  │ (comp_*, strain/stress/deflection, EI/alpha, V_*, V_func_series[4,M])
                                  ▼
run_fram(store, insar, pinn, cfg) ─► FRAMOutput  ◄── 반환값
                                     (CRI[N,M], cri_global_max, warning)
```

| 단계 | 입력 | 출력(핵심) | 다음 단계가 쓰는 것 |
|---|---|---|---|
| CV | cfg(image_h/w, seed) | roi_mask, member_label, grid_density, geometry | grid_density(샘플링 가중), member_label, azimuth |
| InSAR | CVOutput | los[N,M], longitudinal[N,M], member, dates | longitudinal, dates, member(PINN); longitudinal, xyz, member, dates(FRAM) |
| PINN | InSAROutput | comp_thermal/settle/anomaly, V_func_series[4,M] | V_func_series, comp_anomaly/settle/thermal(FRAM) |
| FRAM | InSAROutput + PINNOutput | CRI[N,M], cri_global_max, warning | — (최종 산출물, 대시보드가 소비) |

> FRAM은 InSAR와 PINN **양쪽**을 동시에 받는다: PINN의 `V_func_series`로 공명 R_ij를, InSAR의 변위로 공간/발산 항을 계산하기 때문.

<summary><b>5.5 FRAM 엔진 (PARTIAL + REAL/Phase 5)</b> — 공명 진단/CRI</summary>

  - **A(증폭률)**: 4기능 변동 Vi 스택 → 전역 max 정규화 합 대비 `|d(los)/dt|`. `Vi[1]`(load)이 `thermal gradient * 0.3` **하드코딩 가짜값**.

  - **R_div**: los 시간 2차미분 `|d²/dt²|`.

- **주의점**: `__init__.py`가 비어 풀패스 임포트만 가능. resume 을 안 쓰면(기본) `mode='w'`로 매 실행 기존 project.h5 덮어씀 — 항상 4단계 전부 실행.

## 6. 구현 성숙도 지도

```
REAL    ██████████  contracts, orchestrator, 핫스왑, 골든회귀, **4대 엔진 전부 real**(CV/InSAR/PINN/FRAM)
                    + InSAR 선별 A~E·인제스트 G·inventory
PARTIAL ████        dashboard, tests/docs, InSAR/F 코어(WSL2 템플릿)
STUB    ██          각 엔진 stub 은 핫스왑 기본값으로 병존(real 은 --engine X=real 로 선택)
```

- **이제 4대 엔진 모두 real 구현이 존재한다** — 핫스왑 스위치·골든 회귀를 먼저 깐 뒤 엔진을 하나씩 실구현으로 채웠다:
  - **InSAR**: 선별 A~E(OSM·ASF·ERA5·번들)+인제스트 G(`run_insar_real`) REAL, 코어 F(SARvey)는 `processing.py` demo/real + WSL2 스크립트.
  - **PINN**: PyTorch PINN + Euler-Bernoulli PDE + FEM 모달(해석해 검증).
  - **FRAM**: stub 휴리스틱 결함 3종(load 하드코딩·R_ij broadcast·self-max) 수정한 real.
  - **CV**: 영상처리(Otsu+CC) + Transformer 분할 백엔드.
- **stub 도 병존** — 핫스왑 기본값. 골든 회귀는 stub 데모를 스냅샷하므로 real 추가에 영향 없음.
- 모두 **합성/데모로 검증**(사용자 실데이터 `insar_data` 미사용). 실데이터 정확도는 게이트 G2~G5 에서 별도 검증 대상.

핵심 전환: **"계약은 성역, 엔진은 교체품"** 전략으로 4대 엔진을 stub→real 로 모두 채웠고, 이제 stub/real 을 자유 조합(`--engine X=real`)할 수 있다.

- ✅ `pinn/real_engine.py`: PyTorch PINN + autograd 4차 **Euler-Bernoulli PDE** 손실 + EI 역산(점별, 손상 국소화) + 4성분 분해 + **FEM 모달**(해석해 5%내 검증).
- ✅ V_load 를 PDE 이탈로 재정의. ⚠️ `V_func_series_ds[4,M]` 순서 보존됨.
- ✅ **절대 EI 식별 + 비차원화**(`_identify_EI_from_pde`): 비차원 PDE 균형 `EI=q·L⁴/(w_scale·⟨|∂⁴ŵ/∂x̂⁴|⟩)`(가정 자중 q=Q0)로 스케일 모호성 해소(단순보 해석해 회수 검증). 기존 `log_EI`(gradient 0 → 상수 박힘) 결함 제거. ⚠️ 합성 데이터는 분포하중 휨이 약해 d4≈0 → EI 가 강체 상한으로 포화(실데이터 필요).

- ✅ `fram/real_engine.py`: load 하드코딩 제거(실 comp_load)·R_ij **점별 공명**(broadcast 결함 해소)·self-max→**절대 보정**(sat 함수).
- ✅ **함수망 공명(N-K)**: R_ij 기능 결합 네트워크의 스펙트럼 반경 → 시스템 공명 강도 S(t)[M](`_network_resonance`), 초기 윈도우 신뢰도 램프로 노이즈 억제, 점별 결합항 증폭으로 CRI 반영. (4노드 그래프는 numpy `eigvalsh` — networkx 불필요.)
- ✅ **Morandi 합성 검증(게이트 G5)**: 가속 침하 붕괴 전조 시나리오(`fram/synthetic.make_collapse_scenario`, 라벨 포함) → InSAR→PINN→FRAM 관통 시 CRI 가 failing 점을 **ROC-AUC=0.946(≥0.9)** 로 판별. `roc_auc`(의존성 없음). 전조 초과위험의 시간 심화도 검증.
- ✅ **isotonic 캘리브레이션**(`fram/calibration.py`): PAVA(sklearn 미의존)로 CRI→붕괴확률 단조 매핑 학습(`IsotonicCalibrator`, 직렬화). 단조라 **AUC 보존(0.946→0.961)·Brier 개선(0.234→0.059)**. `cfg.fram_calibrator` 주면 FRAM 이 보정 확률맵 `calibrated_risk[N,M]` 추가 산출(선택, 기본 None → 골든 안전).
- ⬜ 남은 것: FastAPI/Prefect 실시간.

**권장 착수 순서**: Phase 1 인프라(핫스왑·골든회귀) → Phase 3 InSAR(inventory가 이미 출발선) 또는 Phase 2 CV(상류부터). 단방향 흐름상 CV가 최상류이므로 정합성 정합이 중요하면 CV→InSAR 순, 실데이터 가용성이 우선이면 InSAR부터.

## 8. 주의점·리스크

### 계약 깨짐 위험
- **`V_func_series_ds[4,M]` 행 순서**가 `('thermal','load','bearing','foundation')`와 어긋나면 FRAM 공명 계산이 조용히 잘못된다. PINN 교체 시 최우선 보존 대상.
- **HDF5 attribute 64KB 한계**: `CVGeometry.centerline`이 `*_ds` 규약의 예외로 좌표 리스트를 직접 보유 → CV 실구현에서 축선이 길어지면 meta JSON attribute 오버플로 가능.
- **`*_ds` 경로 정합성 미강제**: 잘못된 경로 문자열도 검증 안 됨 → `read_array` 시점 KeyError.
- **`read_meta` 타입 미검증**: `attrs['meta_type']` 교차검증 없음 → 잘못된 model 전달 시 오파싱.

### 가정·연결 끊김
- InSAR engine은 `cv.grid_density_ds`가 None이 아님을 가정(schema상 Optional).
- InSAR engine과 inventory가 **현재 연결 끊겨 있음** — Phase 1/3에서 inventory 메타가 engine 입력으로 흘러들어가야 함.
- FRAM의 모든 지표가 self-max 정규화 → 절대 위험 임계와의 결합 의미가 약함(저변동 데이터에서 가짜 위험).
- `coherence`/`temporal_coherence`가 InSAR에서 동일 배열 중복 → 실데이터에서 분리 필요.

### 테스트 커버리지 공백
- **문서 §1의 "pytest 3/3"과 실제 테스트 5함수(+4 parametrize)가 불일치** — Phase 0 완료 기준이 갱신되지 않았을 가능성. 워크플로우 문서를 신뢰원으로 갱신 권장.
- 테스트는 형상·dtype·물리범위(V∈[0,1], CRI∈[0,1], isfinite)·단조성·계약 불변식 + 4대 엔진 real 의 동작/통합을 **합성·데모로** 검증한다(총 151 테스트). **전(全) real 엔진 통합**(`test_all_real_integration.py`): CV+InSAR+PINN+FRAM real 을 동시에 켜고 geo 좌표 체인(CV geo_transform→world 정합→world xyz→DEM z)·identity 폴백·계약 N/M 전파·V_func 순서·network_resonance 까지 end-to-end 관통 확인. 정확도 게이트도 합성으로 검증됨 — G2(부재 mIoU·축선, `test_cv_metrics.py`)·G5(Morandi ROC-AUC=0.946·isotonic 캘리브, `test_fram_morandi.py`/`test_fram_calibration.py`). **실데이터 수치 게이트(실 mAP/EI오차/실 ROC)는 사용자 실데이터·실 제원·실 가중치가 들어와야 평가 가능.**
- 증분 재개(축 B)는 `tests/test_incremental_resume.py`(11함수)로 검증됨 — fingerprint 사슬·계획(compute/reuse)·핫스왑 무효화·크래시 복구(출력 손상)·force cascade·결정론 동일성. 기본 경로(resume=False, mode='w' 전체 덮어쓰기)는 골든 회귀가 별도 보호.
